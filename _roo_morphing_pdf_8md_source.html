5 BuildRooMorphing function {#BuildRooMorph}
6 =========================
7 
8 **File**: CombinePdfs/src/MorphFunctions.cc
9 
10 The PDF which is used for the signal is a custom RooFit PDF. Its application is
11 made via the datacard production code, with a call to a function named
12 BuildRooMorphing. Here we highlight some of the important features of this
13 function. For a greater level of detail we recommend the reader look at the code
14 itself, which is well commented.
15 
16 The arguments to the function are as follows:
17 
18 \snippet CombinePdfs/src/MorphFunctions.cc part1
19 
20 The options to this function are as follows:
21 
22 * A RooWorkspace to be passed to the function.
23 * The Combine Harvester instance, already setup with all the necessary channels,
24 categories and systematics as described in the previous sections.
25 * The process upon which the morphing is to be applied, each signal
26 process.
27 * A RooAbsReal which controls the signal mass.
28 * The "norm_prefix" is an optional additional prefix to add to the normalisation term
29 controlling the PDF.
30 * The "allow_morph" option controls whether the PDF is
31 designed such that for masses inbetween those for which signal templates are
32 available the signal yield and shape are taken via interpolation, or whether the
33 nearest template is simply taken.
34 * Simply controls the verbose output of the code
35 * The option "force_template_limit" is an optional setting to force the
36 normalisation of the PDF to go to zero outside of the range of available masses.
37 * An optional TFile to contain debug information.
38 
39 The first step of the function is to work out the list of available masspoints
40 for the signal process, and sort and count them:
41 
42 \snippet CombinePdfs/src/MorphFunctions.cc part2
43 
44 Then it is necessary to track all the information on the systematics affecting
45 the signal process. This is done separately for the shape and normalisation
46 systematics. After some manipulation of this information (including special
47 treatment for the case where a shape systematic also alters the normalisation,
48 i.e. has a value different from 1.00), it is possible to build a RooArgList of
49 the parameters controlling the vertical template morphing which is internally
50 used by combine to apply the effect of the shape systematics to the signal.
51 
52 \snippet CombinePdfs/src/MorphFunctions.cc part3
53 
54 Note that the above code takes care over the possibility that the shape
55 systematics are not the same for all masspoints - this is not currently
56 supported in this function. The created "ss_list" is used later to create the
57 vertical template morphing PDFs.
58 
59 For the normalisation systematics we must consider separately the two cases: 1)
60 in the first case the uncertainty is the same for each masspoint, so we can
61 leave this in the datacard in the usual way, but in case 2) where the
62 uncertainty is not the same for each masspoint, we have to include this
63 information in the signal PDF by creating a RooFit object that makes the
64 uncertainty a function off mass. Finally a list is built of all objects required
65 for the interpolation.
66 
67 A 1D spline is built directly from the array of rates and masses:
68 
69 \snippet CombinePdfs/src/MorphFunctions.cc part4
70 
71 Here the option "force_template_limit" allows the user to use an alternative
72 rate array where the rate falls to 0 for masses only very slightly outside of
73 those in the mass array. Then a normalisation term is built which contains all
74 the terms which go into the normalisation - the original rate and the
75 normalisation systematics. The array of vertical morphing PDFs is also built
76 using the information from earlier. Finally, the RooMorphingPdf is built with
77 the following obtained information:
78 
79 \snippet CombinePdfs/src/MorphFunctions.cc part5
80 
81 with the arguments:
82 * xvar : the fixed "CMS_th1x" x-axis variable with uniform integer binning
83 * mass_var: the floating mass value for the interpolation
84 * vpdf_list: the list of vertical-interpolation pdfs
85 * m_vec: the corresponding list of mass points
86 * allow_morph: if false will just evaluate to the closest pdf in mass
87 * data_hist.GetXaxis(): The original (non-uniform) target binning
88 * proc_hist.GetXaxis(): The original (non-uniform) morphing binning
89 
90 The final two arguments are needed to support the possibility that the signal
91 template being interpolated has a finer binning than the target binning for the
92 fit. in order to avoid known problems with the RMS of a peaking distribution
93 not being morphed smoothly from mass point to mass point if the binning is too
94 wide. A RooProduct is added for the normalisation of this PDF:
95 
96 \snippet CombinePdfs/src/MorphFunctions.cc part6
97 
98 The creation of this term is very important. When building the workspace, this
99 term will be searched for and tied to the PDF. It is possible at the workspace
100 building step to add additional terms to those normalisation term, for example a
101 scaling for cross-section times branching ratio. Example usage of this will be
102 discussed later.
103 
104 The final step in the function is to tidy up the CH instance before it is
105 returned to the main code to write out the datacards. The signal yield in the
106 datacards is replaced with a single value of 1.0, such that the normalisation
107 will now be read from the normalisation object just created.

109 Example usage for SM analysis {#SMMorph}
110 =========================
111 
112 The code to produce SM datacards is almost exactly the same as the example
113 described previously with the exception of a few additional lines to call the
114 BuildRooMorphing function. The call to BuildRooMorphing is as follows:
115 
116 \snippet CombinePdfs/bin/SMLegacyMorphing.cpp part1
117 
118 A single call is made for each signal process of each channel and category. The
119 produced workspace is then imported into the CH instance and the created PDFs
120 extracted. The workspace is stored in the shape file created for the limit
121 calculations alongside the text datacards.
122 
123 For cases like the SM analysis, this method can be used simply to produce one
124 combined workspace for all masses, making the mass a continous parameter of the
125 workspace. The rest of the calculations of the statistical results on the
126 workspace are unchanged, except for the requirement to tell combine to fix the
127 mass rather than float it. The workspace is build using the command:
128 
129 text2workspace.py -b combinedCard.txt -o combinedCard.root
130  --default-morphing shape2
131 
132  Note that mass is not specified here, since we want to build a combined
133  workspace for all possible masses. Then, to run the maximum likelihood fit for a
134  given mass:
135 
136  combine -M MaxLikelihoodFit --freezeNuisances MH -m 125 combinedCard.root
137 
138  where in this case the mass must be specified.

156 Applying a model at text2workspace step {#MSSMModel}
157 =========================
158 
159 Having produced datacards containing signal PDFs for the appropriate number of
160 Higgs bosons, we can go one step further and make extra use of the normalisation
161 term associated with the PDFs. One specific use of this, as discussed in the
162 introduction, is in the model dependent exclusions provided for example in the
163 MSSM H->tautau analysis. In this case, each point in mA-tanb
164 phase space has a different signal model containing contributions from the three
165 Higgs bosons. At each point the masses of the Higgs bosons and the
166 cross-sections times branching ratios are different. The normal procedure in
167 this case would be to produce a workspace for each model point with a specific
168 signal template built and added by hand.
169 
170 Now that we have RooMorphingPdfs we can do this in a more clever way, by
171 attaching terms controlling the normalisation of the signal to its normalisation
172 term. This can be done within the text2workspace.py step, where the .txt
173 datacard is converted into a RooWorkspace as input to combine. In this step, it
174 is possible to apply a physics model which performs some scaling and
175 manipulation to the different quantities in the datacards. By designing an
176 appropriate physics model, we can add terms to the normalisation to control the
177 cross section times branching ratio of the three Higgs bosons.
178 These can be added as a function of mA and tanb which can be
179 made parameters of the workspace as RooRealVars:
180 
181 \snippet CombinePdfs/python/MSSM.py part1
182 
183 The histograms containing the information on the different cross-sections and
184 branching ratios are included as TH2Ds inside a file for a particular model.
185 These are read in and turned into a RooHistFunc via a helper function called
186 doHistFunc:
187 
188 \snippet CombinePdfs/python/MSSM.py part2
189 
190 Note the option 'interpolate' which is off by default - this allows the
191 possibility of interpolating the model inputs to reach values beyond the
192 granularity that is provided.
193 
194 RooHistFuncs are then built for the mass terms controlling the two Higgs bosons
195 which are not the A boson (since this is already controlled by the parameter
196 mA, as well as the cross-section and branching ratio, for example using
197 terms:
198 
199 \snippet CombinePdfs/python/MSSM.py part3
200 
201 In this code snippet, 'era' controls the centre of mass energy of the model,
202 currently either 7TeV, 8TeV or 13TeV. Note that theoretical uncertainties,
203 which are also provided from the model inputs as a function of mA and
204 tanb, are applied at this stage as well by building the relevant terms
205 and adding them to the workspace. Finally the relevant parameters are combined
206 to give a complete term used to control the normalisation of the signal.
